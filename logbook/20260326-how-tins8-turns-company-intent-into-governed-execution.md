# Control-Plane Architecture For Intent, Reconciliation, And Sandboxed Execution

The Founders Control Plane should be built around a control loop rather than around a conversation loop. A conversation is useful for interacting with the system, but it is not a durable operating model. The system needs persistent intent, persistent policy, persistent observed state, and a reconciler that keeps comparing those objects until the underlying goal is satisfied, blocked, replaced, or explicitly abandoned.

That distinction is important because many AI systems start from the worker and try to add structure later. The result is usually a capable runtime with weak operational memory. It can explain what it plans to do next, but it does not reliably own the surrounding state model. A control plane has to work the other way around. It owns the contract first and treats workers, tools, and models as implementation details inside that contract.

Infrastructure provisioning is again the easiest place to make this concrete. Imagine that the company wants a new isolated preview environment for enterprise demos. In a conventional automation setup, that might become a workflow: open request, request approval, run Terraform, notify owner. In a control-plane architecture, the system has to represent more than the sequence. It has to represent the desired state itself. Which kind of environment is being requested? In which account or project should it live? Which network shape is allowed? Which budget tier applies? Which identities may administer it? Which secrets may be mounted? Which policies must hold before the environment is considered valid?

Once that desired state exists explicitly, reconciliation becomes the core mechanism. The system compares the declared state with observed reality and keeps driving the gap down. If the environment does not exist, the gap is obvious. If it exists but in the wrong account, with the wrong ingress policy, without an owner, or above the expected cost profile, the gap is still there. That is the key advantage of thinking in control-plane terms: the system is not only responsible for getting work started, but for evaluating whether the resulting state actually matches intent.

From that perspective, the product-defining layer is not the worker runtime. It is the set of core objects around the runtime. The system needs durable representations for intent, observation, policy, approval, capability, evidence, and reconciler status. Those objects should exist independently of whichever model or tool is used to process them. If the system only remembers intent indirectly through a prompt transcript or a previous agent run, it is not yet acting as a control plane.

This is also where bounded execution matters. The useful unit is not one monolithic autonomous agent. It is a small worker with a narrow context, an explicit scope of action, a defined execution environment, and an output that fits back into the system model. One worker might analyze cost signals and summarize infrastructure drift. Another might propose the child resources needed for a new environment. Another might perform a deterministic policy check. Another might run an isolated execution task. A human review step can be treated as just another bounded node in the same loop. What matters is that each of these steps stays legible and that the system, not the worker, owns the long-lived state.

There is a useful analogy here to reconcile-driven infrastructure control planes. In that style of system, the contract remains declarative and the reconcile loop remains authoritative, even if custom logic is used to derive child state. The same discipline applies here. Agent-backed logic can help assemble plans, interpret observed state, summarize logs, or derive a set of proposed actions, but it should sit inside a bounded step with schema-constrained outputs. The system boundary should remain deterministic even when some internal reasoning is probabilistic.

That is why planning and execution should be separated. A reasoning step should usually produce a proposal, a plan, or an execution request. It should not automatically become the privileged runtime that performs the action. In the provisioning example, the planning step might infer that the requested preview environment fits an approved golden path, requires a cost cap, and should inherit a standard network policy. But the actual provisioning action should happen through a separate execution path after policy evaluation. Otherwise the planning layer quietly turns into the thing with real standing authority.

Sandboxing enters at exactly that point. Once the system can run tools, generated code, infrastructure clients, or repository operations, the execution substrate becomes part of the architecture. The executor should run in an isolated environment with short-lived credentials, restricted network access, explicit tool allowlists, and durable artifacts. That is not an implementation nicety. It is part of the trust model. If a worker is allowed to provision infrastructure, inspect repositories, or call external APIs, the system should be able to show which capabilities were granted for that run and prove that the run was confined to those capabilities.

Returning to the provisioning example makes the separation clearer. A founder requests a new demo environment. The control plane records that as desired state. A bounded planning step assembles context: environment type, owner, budget class, target cloud account, required policies, and related observations from the existing estate. That step may use model-assisted reasoning to classify the request or synthesize a rollout proposal, but it emits a structured execution request rather than directly mutating infrastructure. A policy layer then checks whether the request fits an allowed envelope. If it does, a sandboxed executor receives only the credentials and tools needed to perform that one provisioning task. When the run finishes, artifacts and observed state flow back into the control plane. The reconciler then decides whether the intent is satisfied or whether a mismatch remains.

That shape is much closer to what a real company operating layer needs. It lets the system be flexible in the middle while staying strict at the boundary. It also avoids the failure mode where "agentic" becomes a euphemism for "the system took action but nobody can explain under what contract or with which authority."

The architectural rule is simple. The worker is not the system. The system owns intent, state, policy, and provenance. Workers only help move those objects toward alignment. If that rule is preserved, model-assisted reasoning can be introduced without giving up control-plane discipline. If that rule is lost, the product collapses back into an execution tool with better language output.

For a first implementation, that is already enough. The system does not need to solve every kind of company coordination on day one. It needs to prove that intent can be modeled durably, that infrastructure provisioning can be reconciled against policy and observed state, and that execution can be delegated to bounded workers without breaking the trust model. If those pieces hold, the broader operating layer can be built on top of them.
